1 Acceso al Panel de Admin

El panel de administración de Django está disponible en /admin/. Requiere que el usuario tenga el flag is_staff = True. Los superusuarios pueden ver y modificar todos los registros; los usuarios staff ven solo lo que sus permisos de grupo permiten.

Autenticación Local

Usuario y contraseña creados con python manage.py createsuperuser.

Verificación de contraseña gestionada por Django directamente.

Autenticación LDAP

Los usuarios LDAP reciben is_staff / is_superuser según su grupo en LDAP Group Maps.

Su contraseña siempre es "no usable" — el acceso es vía directorio LDAP.

⚠ Si un usuario LDAP pierde acceso (AD desconectado), un superusuario local debe otorgar el acceso temporalmente desde Admin → Users.

2 Configuración LDAP

Todo lo relacionado con autenticación contra Active Directory se configura desde el admin — sin necesidad de editar archivos .env ni reiniciar el servidor. Los cambios toman efecto en el siguiente intento de login (caché de 60 segundos).

LDAP Configuration

Admin → Lb Manager → LDAP Configurations

Define el servidor LDAP al que se conecta la aplicación. Solo puede existir un registro activo.

CampoTipoDescripción
server_uriTextoURI del servidor LDAP. Ej: ldap://dc01.empresa.local o ldaps://dc01.empresa.local
use_tlsBooleanoActiva STARTTLS sobre la conexión LDAP (puerto 389 con cifrado). No usar si el URI ya es ldaps://.
network_timeoutEntero (seg)Tiempo máximo de espera por conexión. Default: 10 segundos.
activeBooleanoSolo el primer registro activo es usado. Desactivar deshabilita LDAP sin borrarlo.
ℹ Si la tabla está vacía, el backend cae al fallback de settings.py (AUTH_LDAP_SERVER_URI).

LDAP Group Maps

Admin → Lb Manager → LDAP Group Maps

Mapea un contenedor LDAP (OU) a un Grupo de Django. Cuando un usuario inicia sesión, el backend verifica en qué OU vive su cuenta en el directorio y asigna el grupo correspondiente automáticamente.

CampoTipoDescripción
orderEnteroOrden de evaluación. Los registros con número menor se evalúan primero.
container_dnTextoDN del contenedor LDAP donde viven los usuarios de este grupo. Ej: OU=NetworkOps,DC=empresa,DC=local
dn_templateTextoPlantilla del DN del usuario. Use %(user)s como placeholder. Ej: CN=%(user)s,OU=NetworkOps,DC=empresa,DC=local
django_groupFK → GroupGrupo de Django que se asigna al usuario si coincide con este contenedor.
grants_superuserBooleanoSi está activo, el usuario recibe is_superuser = True al iniciar sesión.
login_redirect_urlTextoURL a donde redirigir al usuario de este grupo después del login. Debe comenzar con /. Vacío = dashboard predeterminado.
activeBooleanoDesactivar omite esta fila sin borrarla.
⚠ Sincronización de grupos: En cada login, todos los grupos gestionados son revisados. Si el usuario ya no está en el OU correcto, se le quita el grupo automáticamente.

4 CSV — Importación de Datos

El sistema de importación CSV tiene dos componentes complementarios: CSV Table Upload Configs registra qué tablas pueden recibir datos; CSV Import Configs define los jobs programados que leen un archivo del servidor y lo importan automáticamente. Además, cualquier tabla registrada puede recibir carga manual desde el menú Datos → CSV Upload.

4a. CSV Table Upload Configs — Registrar tablas

CSV Table Upload Configs

Admin → Lb Manager → CSV Table Upload Configs

Paso 1: registrar qué modelos Django pueden recibir datos via CSV. Una vez registrada y activa, la tabla aparece disponible en CSV Import Configs (jobs programados) y en la UI de carga manual.

CampoTipoDescripción
labelTextoNombre visible en la UI. Ej: "LB Hardening".
modelSelectorModelo Django destino. El dropdown lista todos los modelos del proyecto automáticamente.
unique_fieldsTexto (opcional)Campos separados por coma usados como clave de deduplicación para update_or_create. Ej: device,fecha. Si se deja vacío, cada fila del CSV se inserta directamente sin verificar duplicados.
activeBooleanoDesactivar oculta la tabla de la UI y de los selectores de jobs, sin borrarla.
descriptionTextoNotas opcionales sobre el formato esperado o nombres de columnas.

4b. CSV Import Configs — Jobs programados

CSV Import Configs

Admin → Lb Manager → CSV Import Configs

Paso 2: definir el job. Referencia una tabla registrada, indica la ruta del archivo y la programación cron. Al guardar, el Schedule de django-q2 se crea o actualiza automáticamente.

CampoTipoDescripción
nameTextoNombre descriptivo del job. Ej: "F5 Healthcheck Diario".
typeSelectorTabla destino — muestra las CSV Table Upload Configs activas por su label.
file_path_templateTextoRuta absoluta al archivo CSV. Usa {date} para la fecha. Ej: /data/lb/healthcheck_{date}.csv
date_formatTextoFormato Python para {date}. Default: %Y-%m-%d → 2025-06-01.
cron_scheduleTextoExpresión cron de 5 campos. Ej: 0 6 * * * = todos los días a las 6:00 AM.
activeBooleanoDesactivar pausa el job. El Schedule de django-q2 se elimina automáticamente.
last_run_atSolo lecturaFecha/hora de la última ejecución.
last_run_statusSolo lecturanever / ok / error
last_run_messageSolo lecturaResumen: inserted=N, updated=N, errors=N o mensaje de error.
Expresión cronDescripción
0 6 * * *Todos los días a las 6:00 AM
30 5 * * 1-5Lunes a viernes a las 5:30 AM
0 7 * * 1Todos los lunes a las 7:00 AM
0 */6 * * *Cada 6 horas
*/30 * * * *Cada 30 minutos
ℹ Si la tabla tiene unique_fields, la importación usa update_or_create: los registros existentes se actualizan y los nuevos se insertan. Si unique_fields está vacío, cada fila se inserta directamente (útil para tablas sin clave natural única).

Mapeo de columnas (Column Mappings)

Al editar un CSV Import Config, la sección Column Mappings permite mapear nombres de columna del CSV a campos del modelo. El selector Model field se puebla automáticamente con los campos del modelo seleccionado en Type.

Si no defines ninguna fila, los headers del CSV se comparan automáticamente con los nombres de campo del modelo (sin distinguir mayúsculas). Solo es necesario agregar mappings cuando los nombres difieren.

La conversión de tipo (int, float, bool, date, datetime, str) se aplica automáticamente según el tipo de campo del modelo.

Para ejecutar manualmente: en la lista de CSV Import Configs, marcar el checkbox → acción "▶ Run now" → Go. El resultado aparece en last_run_status una vez que el worker procesa la tarea.

⚠ El worker (python manage.py qcluster) debe estar activo para procesar tareas. Si no está corriendo, la tarea queda en cola.

Carga manual desde la UI

Menú → Datos → CSV Upload

Cualquier tabla registrada como activa en CSV Table Upload Configs aparece en Datos → CSV Upload. Desde ahí cualquier usuario con permiso view_csvtableuploadconfig puede subir un archivo CSV manualmente.

  1. Ir a Datos → CSV Upload y seleccionar la tabla deseada.
  2. Subir el archivo .csv (UTF-8, primera fila = encabezado).
  3. El sistema muestra el resultado: filas insertadas, actualizadas, omitidas y errores.

Los headers del CSV se comparan automáticamente con los campos del modelo (sin distinguir mayúsculas). No requiere configuración adicional si los nombres coinciden.

5 Script Run Configs — Ejecución de Scripts

Permite configurar y ejecutar scripts Python externos (sincronización de VIPs, Pools, SSL Certs, etc.) directamente desde el Admin. El script recibe la fecha como primer argumento y su salida (stdout + stderr) queda registrada en el Admin para diagnóstico.

Script Run Configs

Admin → Lb Manager → Script Run Configs
CampoTipoDescripción
nameTextoNombre único del job. Ej: "Sync F5 Inventory".
descriptionTextoNotas opcionales sobre qué hace el script.
script_pathTextoRuta absoluta al script Python dentro del container. Ej: /app/scripts/sync_inventory.py
run_dateFechaFecha que se pasa como primer argumento al script. Por defecto: hoy. Cambiar este valor antes de usar "Run now" para procesar una fecha diferente.
date_formatTextoFormato strftime para convertir run_date a texto. Default: %Y-%m-%d → 2026-03-13.
cron_scheduleTextoExpresión cron de 5 campos. Dejar vacío para deshabilitar la programación automática. Ej: 0 6 * * * = todos los días a las 6:00 AM.
activeBooleanoDesactivar pausa el job sin borrarlo. Si también hay cron_schedule, el Schedule de django-q2 se elimina automáticamente.
last_run_atSolo lecturaFecha/hora de la última ejecución.
last_run_statusSolo lecturapending / ok / error
last_run_messageSolo lecturaexit_code del proceso y hasta 4 000 caracteres de stdout + stderr. Útil para diagnosticar fallas sin acceder al servidor.
ℹ El script se ejecuta como: python <script_path> <run_date> usando el mismo intérprete Python del container. El script puede usar los paquetes ya instalados en el virtualenv de la app (requests, psycopg2, etc.).

Cómo crear un nuevo Script Config

  1. Ir a Admin → Lb Manager → Script Run Configs → Agregar.
  2. Completar el Name (ej: "Sync F5 VIPs y Pools").
  3. Ingresar el Script path — ruta absoluta al .py dentro del container (ej: /app/scripts/sync_f5_inventory.py).
  4. Ajustar Run date a la fecha que necesitas procesar (por defecto es hoy).
  5. Si el script espera un formato de fecha distinto a YYYY-MM-DD, ajustar Date format.
  6. Opcionalmente definir Cron schedule para ejecución automática.
  7. Guardar — si tiene cron_schedule, el Schedule se crea automáticamente en Django Q → Scheduled Tasks.
⚠ El script_path debe existir dentro del container. Si el script está en un directorio montado como volumen, verifica que el bind mount esté configurado en docker-compose.yml.

Ejecutar manualmente (Run now)

  1. Editar el Script Config y cambiar Run date a la fecha deseada. Guardar.
  2. En la lista de Script Run Configs, marcar el checkbox del script deseado.
  3. En el menú desplegable seleccionar "▶ Run now (enqueue immediately)". Hacer clic en Go.
  4. Una vez procesado, revisar Last run status y Last run message para ver el resultado y la salida del script.
ℹ Si last_run_status muestra error, revisar Last run message — contiene el exit_code y el traceback. También se puede ver más detalle en Django Q → Failed Tasks.

7 Health Rules — Reglas de Alerta Automática

Definen umbrales sobre los campos del health check F5. Cuando el comando evaluate_health_rules corre (vía django-q2 o manualmente), compara los últimos snapshots contra estas reglas y abre tickets en la Bitácora automáticamente.

Health Rules

Admin → Lb Manager → Health Rules
CampoTipoDescripción
field_nameOpciónCampo del modelo HealthCheckF5 a evaluar. Ej: cpu_usage, nodes_down.
operatorOpciónOperador de comparación: >, >=, <, <=, ==, !=.
thresholdDecimalValor umbral. Si la condición campo operador threshold es verdadera, se genera un ticket.
severityOpciónSeveridad del ticket generado: LOW, MEDIUM, HIGH, CRITICAL.
✔ Deduplicación automática: Si ya existe un ticket abierto para el mismo dispositivo y regla, no se crea un duplicado. Solo se abre uno nuevo cuando el anterior está cerrado o resuelto.

Ejemplo de reglas útiles:

CampoOperadorThresholdSeveridadSignificado

nodes_down>0MEDIUMAl menos 1 nodo caído

9b Certificate Health Checks

Vista de revisión diaria de certificados SSL/TLS recopilados desde dispositivos de red. Los datos se cargan mediante un script Python que lee un CSV generado por Ansible.

Health Check Certificate

/health/certificates/

Muestra una tabla con los certificados recolectados, incluyendo su fecha de vencimiento y días restantes. Los días se colorean automáticamente:

  • Rojo — menos de 30 días para vencer (acción urgente).
  • Amarillo — entre 30 y 59 días (atención próxima).
  • Verde — 60 días o más (estado normal).
CampoTipoDescripción
deviceTextoNombre o FQDN del dispositivo que contiene el certificado.
expiration_dateTextoFecha de vencimiento tal como la reporta el dispositivo.
days_remainingEnteroDías que faltan para el vencimiento al momento de la recolección.
certificate_typeTextoTipo de certificado (server, CA, etc.).
commentsTextoObservaciones adicionales del script de recolección.
alternamesTextoSubject Alternative Names (SANs) del certificado.
fechaFechaFecha en que se realizó la recolección (CURDATE del script).
companyTextoCliente o empresa asociada al dispositivo (opcional).

Filtros disponibles

  • Fecha desde / hasta — rango de fechas de recolección.
  • Device (FQDN) — búsqueda parcial por nombre de dispositivo.
  • Days Max — muestra solo certificados con days_remaining ≤ N. Útil para ver todos los certs que vencen en los próximos 30, 60 o 90 días.

Nota para el script de carga: La tabla PostgreSQL se llama health_check_certificate. El campo CSV Alternames se almacena como alternames (minúsculas). A diferencia de las tablas F5-sync, no se hace TRUNCATE antes de insertar — los registros se acumulan por día.

15 Database Backups

Permite crear, descargar y eliminar backups de la base de datos PostgreSQL en formato pg_dump custom (.dump). Accesible desde el menú de staff: Admin → Database Backups. La ruta, retención y schedule se configuran en Site Settings.

Crear backup manual

Admin → Database Backups → Crear backup
  1. Ir a Admin → Database Backups.
  2. Hacer clic en Crear backup ahora.
  3. El archivo .dump se guarda en el directorio configurado en backup_path (Site Settings).
  4. Los backups más antiguos que backup_retention_days se eliminan automáticamente.
⚠ Requiere que pg_dump esté instalado. Si no está en el PATH, configurar la ruta en Site Settings → backup_pg_dump_path.

Gestión de archivos

La lista muestra todos los archivos .dump en el directorio configurado, ordenados por fecha de creación (más reciente primero).

AcciónDescripción
DescargarDescarga el archivo .dump directamente al navegador.
EliminarBorra el archivo del servidor. Requiere confirmación.

Restaurar un backup:

pg_restore -d <db_name> -U <user> archivo.dump

Backup automático (schedule)

Configurar en Site Settings → backup_schedule (expresión cron). El worker django-q2 ejecutará python manage.py backup_db según esa programación.

También se puede ejecutar manualmente desde la terminal:

python manage.py backup_db

SSL Dashboard y VIPs con Certificados Vencidos

Dashboard SSL — /ssl/dashboard/

Vista consolidada del estado de los certificados SSL activamente en uso. Solo se muestran certs que están vinculados a través de la cadena completa:

VIP (profiles JSON, context="client-side")
  → ClientSSLProfile (mismo ltm_fqdn)
    → SSLCert (full_path = certificate_file, mismo ltm_fqdn)

Los certificados sin VIP asociada no aparecen en este dashboard.

SemáforoCondición
VencidoCertificado con fecha de expiración pasada
CríticoVence en ≤ 30 días
AvisoVence en 31–90 días
OKMás de 90 días restantes

El contador Total Certificados refleja únicamente los certs que llegan al final de la cadena, no todos los certs instalados en los F5.

VIPs Certificados Vencidos / Por Vencer — /ssl/vips-expired/

Lista de VIPs cuyo certificado ya venció o vence en los próximos 30 días. Cada tarjeta muestra:

CampoDescripción
VIP NameNombre de la VIP afectada y su dispositivo F5
DestinationIP y puerto del servicio
SSL ProfileNombre del ClientSSLProfile que aplica el certificado
CertificadoRuta del certificado en el F5 y su fecha de expiración
SubjectSubject del certificado (CN=, O=) — identifica el dominio/entidad cubierta

El color de la tarjeta indica el estado: rojo = ya venció, amarillo = vence pronto. Usa los botones CSV / Excel del encabezado para exportar la lista.

Requisito de datos — cadena VIP → Perfil → Certificado

Para que el Dashboard SSL muestre datos correctos, los tres eslabones deben existir en la BD:

  1. La VIP debe tener en su campo profiles (JSON) una entrada con context = "client-side" que referencie el nombre del ClientSSLProfile.
  2. El ClientSSLProfile debe tener certificate_file igual al full_path del certificado (ej. /Common/cert-000001).
  3. Ambos — perfil y cert — deben tener el mismo ltm_fqdn.

Al poblar datos de prueba, usar python manage.py seed_extra que genera este chain correctamente.

17 REST API v1

La API REST permite integrar la plataforma con herramientas externas, scripts y frontends desacoplados. Todos los endpoints requieren autenticación JWT. Los permisos son los mismos grupos Django usados en la interfaz web.

ℹ URL base: /api/v1/ — disponible en la misma instancia que el panel web.

Autenticación JWT

Obtener tokens de acceso

POST /api/v1/token/

Enviar credenciales de usuario (las mismas que se usan para entrar al panel web):

# Solicitar tokens
curl -X POST https://<servidor>/api/v1/token/ \
     -H 'Content-Type: application/json' \
     -d '{"username": "mi_usuario", "password": "mi_clave"}'

# Respuesta
{
  "access":  "eyJhbGciOiJIUzI1NiIs...",   ← válido 15 minutos
  "refresh": "eyJhbGciOiJIUzI1NiIs..."    ← válido 24 horas
}

Usar el access token en el header de cada petición:

curl https://<servidor>/api/v1/health/f5/ \
     -H 'Authorization: Bearer <access_token>'

Renovar el access token

POST /api/v1/token/refresh/

Cuando el access token expire (15 min), renovarlo sin pedir contraseña de nuevo:

curl -X POST https://<servidor>/api/v1/token/refresh/ \
     -H 'Content-Type: application/json' \
     -d '{"refresh": "<refresh_token>"}'

El refresh token se invalida tras cada uso — se recibe uno nuevo en la respuesta (rotación automática). Un refresh token no usado caduca a las 24 horas.

⚠ Guardar el refresh token de forma segura. Si se pierde o compromete, el usuario debe volver a autenticarse con usuario y contraseña. Los tokens comprometidos pueden invalidarse desde Admin → Token Blacklist → Blacklisted Tokens.

Probar la API desde el navegador

Cómo usar Swagger UI:

  1. Obtener un token: POST /api/v1/token/ (desde curl o desde Swagger mismo)
  2. Abrir /api/schema/swagger/ en el navegador

  4. Ingresar el valor: Bearer <access_token>
  5. Clic en Authorize → Close
  6. Expandir cualquier endpoint y clic en Try it out → Execute

Guía de uso rápido

Ejemplos de uso con curl

# 1. Listar snapshots F5 del día
curl "https://<servidor>/api/v1/health/f5/?fecha=2025-06-01" \
     -H 'Authorization: Bearer <token>'

# 2. Buscar checks de hardening fallidos para un device
curl "https://<servidor>/api/v1/hardening/checks/?device=bigip-01&resultado=failed" \
     -H 'Authorization: Bearer <token>'

# 3. Ver tickets de hardening abiertos
curl "https://<servidor>/api/v1/hardening/tickets/?status=OPEN" \
     -H 'Authorization: Bearer <token>'

# 4. Crear un snapshot DNS (POST)
curl -X POST https://<servidor>/api/v1/health/dns/ \
     -H 'Authorization: Bearer <token>' \
     -H 'Content-Type: application/json' \
     -d '{"fqdn":"dns-01.local","fecha":"2025-06-01","uptime":99.9,...}'

# 5. Cerrar un ticket de hardening (PATCH parcial)
curl -X PATCH https://<servidor>/api/v1/hardening/tickets/42/ \
     -H 'Authorization: Bearer <token>' \
     -H 'Content-Type: application/json' \
     -d '{"status":"CLOSED"}'
⚠ Permisos requeridos: el usuario de API debe tener los mismos permisos Django que en la interfaz web. Un usuario sin view_healthcheckf5 recibirá 403 Forbidden aunque su token sea válido. Asignar los permisos desde Admin → Users → Permissions o mediante grupos.
